## 🎯 核心需求

在顶部工具栏右侧添加一个输入框，支持用户输入自定义敏感词，并使用**逻辑表达式**进行匹配。

---

## 📝 详细需求说明

### 1. UI 需求

**位置**：顶部工具栏右侧  
**组件**：文本输入框  
**占位符提示**：例如 "输入敏感词（支持 !、()、&、|）"  
**样式**：与其他工具栏元素保持一致

---

### 2. 功能需求

#### 2.1 基本功能

- 用户可以在输入框中输入自定义敏感词
- 扫描时，将用户输入的文本作为额外的敏感词规则进行匹配
- 匹配结果与系统内置的敏感词规则合并显示

#### 2.2 逻辑表达式支持

支持以下逻辑运算符：

| 运算符 | 含义 | 示例 | 说明 |
|--------|------|------|------|
| `!` | 非（NOT） | `!密码` | 不包含"密码" |
| `&` | 与（AND） | `密码 & 身份证` | 同时包含"密码"和"身份证" |
| `\|` | 或（OR） | `密码 \| 身份证` | 包含"密码"或"身份证" |
| `()` | 分组 | `(密码 \| 身份证) & 机密` | 先计算括号内，再与"机密"做与运算 |

#### 2.3 优先级规则

**优先级从高到低**：
1. `()` - 括号分组（最高优先级）
2. `!` - 非运算
3. `&` - 与运算
4. `\|` - 或运算（最低优先级）

**示例**：
```
!密码 & (身份证 | 银行卡)
```

**解析顺序**：
1. 先计算 `(身份证 | 银行卡)` → 结果为 A
2. 再计算 `!密码` → 结果为 B
3. 最后计算 `B & A` → 最终结果

### 3. 技术实现要点

#### 3.1 表达式解析器

**重要决策：优先使用成熟的表达式解析库**

为避免重复造轮子并确保解析器的稳定性和正确性，**强烈建议使用成熟的开源库**来实现逻辑表达式解析。

**推荐的开源库**：

1. **expr-eval** (推荐 ⭐⭐⭐⭐⭐)
   - GitHub: https://github.com/silentmatt/expr-eval
   - NPM: `npm install expr-eval`
   - 优点：
     - 轻量级（~5KB gzipped）
     - 支持自定义运算符和函数
     - 完善的错误处理
     - TypeScript 支持
     - 活跃维护（最后更新 < 1年）
   - 适用场景：需要完整的表达式求值功能

2. **boolean-expression-parser**
   - NPM: `npm install boolean-expression-parser`
   - 优点：
     - 专为布尔表达式设计
     - 支持 AND、OR、NOT 运算符
     - 生成 AST
   - 缺点：社区活跃度较低

3. **mathjs** (功能强大但较重)
   - NPM: `npm install mathjs`
   - 优点：
     - 功能极其丰富
     - 支持复杂的数学和逻辑运算
     - 完善的文档和社区
   - 缺点：体积较大（~500KB），可能引入不必要的依赖

**实施建议**：

**方案 A：使用 expr-eval（推荐）**

const parser = new Parser();

// 配置自定义运算符
parser.consts = {};
parser.functions = {
  // 可以添加自定义函数，如关键词匹配
  contains: (text: string, keyword: string) => text.includes(keyword)
};

 * @param context 上下文对象（包含文本和关键词）
 */
export function evaluateExpression(
  expression: string,
  context: { text: string; keywords: Record<string, boolean> }
): boolean {
  try {
    // 将表达式转换为 expr-eval 支持的格式
    // 例如: "密码 & (身份证 | 银行卡)" 
    // 转换为: "keywords['密码'] && (keywords['身份证'] || keywords['银行卡'])"
    const transformedExpr = transformExpression(expression);
    
    const result = parser.evaluate(transformedExpr, context);
    return Boolean(result);
  } catch (error: any) {
    throw new Error(`表达式解析失败: ${error.message}`);
  }
}

/**
 * 转换表达式格式
 */
function transformExpression(expression: string): string {
  // 实现逻辑：
  // 1. Tokenize 表达式
  // 2. 识别关键词
  // 3. 转换为 keywords['xxx'] 格式

**方案 B：自行实现（仅在学习或特殊需求时考虑）**

如果现有库无法满足需求，可以自行实现，但需要：
1. 参考成熟的算法（如 Shunting-yard 算法、递归下降解析器）
2. 编写全面的单元测试
3. 进行充分的边界情况测试
4. 性能基准测试

**自行实现的组件**：
- **Tokenizer**: 词法分析器，将字符串分解为 token
- **Parser**: 语法分析器，构建 AST
- **Evaluator**: 求值器，遍历 AST 并计算结果

---

需要实现一个**逻辑表达式解析器**，包括：

##### 3.1.1 词法分析（Tokenizer）
- 将输入字符串分解为 token（关键词、运算符、括号）
- 识别：普通文本、`!`、`&`、`\|`、`(`、`)`

##### 3.1.2 语法分析（Parser）
- 将 token 序列转换为**抽象语法树（AST）**
- 遵循优先级规则构建正确的树结构

##### 3.1.3 求值器（Evaluator）
- 遍历 AST，对每个节点进行匹配判断
- 返回最终的布尔结果（是否匹配）

#### 3.2 AST 节点类型

```typescript
type ASTNode = 
  | { type: 'keyword'; value: string }          // 关键词节点
  | { type: 'not'; child: ASTNode }              // 非运算节点
  | { type: 'and'; left: ASTNode; right: ASTNode }  // 与运算节点
  | { type: 'or'; left: ASTNode; right: ASTNode };  // 或运算节点
```

#### 3.3 匹配逻辑

对于文件内容中的每一行（或整个文本）：

```typescript
function evaluate(node: ASTNode, text: string): boolean {
  switch (node.type) {
    case 'keyword':
      return text.includes(node.value);
    
    case 'not':
      return !evaluate(node.child, text);
    
    case 'and':
      return evaluate(node.left, text) && evaluate(node.right, text);
    
    case 'or':
      return evaluate(node.left, text) || evaluate(node.right, text);
  }
}
```

#### 3.4 执行环境设计

**方案：表达式解析器在 Worker 线程中执行**

**原因**:
1. 文件内容已经在 Worker 线程中，避免跨进程传输大量文本
2. 匹配计算密集，适合在 Worker 中并行执行
3. 与现有的 `sensitive-detector.ts` 保持一致

**重要性能要求**：
⚠️ **一次文件读取完成所有检测**
- 自定义表达式检测必须与内置规则检测在同一轮扫描中完成
- 禁止多次读取文件或多次扫描同一段文本
- 实现方式：
  1. 使用 `getHighlights()` 一次性扫描内置规则
  2. 使用 `evaluateCustomExpressionOnly()` 评估自定义表达式（不重复扫描内置规则）
  3. 在流式处理的第一个 chunk 上评估表达式（避免重复）

**性能对比**：
- ❌ 错误做法：调用 `countSensitiveMatches()` 会重复扫描内置规则（2次完整扫描）
- ✅ 正确做法：`getHighlights()` + `evaluateCustomExpressionOnly()` （1次扫描+轻量评估）
- 性能提升：减少约 50% 的文本扫描开销

**实现方式**:
1. 将 `expression-parser.ts` 放在 `src/utils/` 目录
2. Worker 线程通过 `import` 引入解析器
3. 主进程将 `customExpression` 通过 `postMessage` 传递给 Worker

**消息传递格式**:
```typescript
// 主进程 → Worker (file-worker.ts)
worker.postMessage({
    taskId: number,
    filePath: string,
    customExpression: string | undefined,  // 新增字段
    enabledSensitiveTypes: string[],
    config: {
        maxFileSizeMb: number,
        maxPdfSizeMb: number
    }
});

// Worker 内部处理流程
async function processTask(task: WorkerTask): Promise<void> {
    const { customExpression, ...otherParams } = task;
    
    // 读取文件内容
    const content = await readFileContent(filePath);
    
    // 执行内置规则检测
    const builtinResults = detectSensitiveData(content, enabledSensitiveTypes);
    
    // 执行自定义表达式检测（如果提供）
    if (customExpression && customExpression.trim()) {
        try {
            const ast = parseExpression(customExpression);
            const isMatch = evaluate(ast, content);
            if (isMatch) {
                builtinResults['custom_expression'] = 1;
            }
        } catch (error) {
            // 表达式解析失败，记录日志但不影响其他检测
            parentPort?.postMessage({
                type: 'warning',
                message: `自定义表达式解析失败: ${error.message}`
            });
        }
    }
    
    // 返回结果
    parentPort?.postMessage({
        type: 'result',
        taskId,
        sensitiveCounts: builtinResults,
        highlights: getHighlights(content, enabledSensitiveTypes)
    });
}
```

#### 4.2 扫描流程集成方案

**集成位置**: `src/detection/sensitive-detector.ts`

**方案 A：扩展现有函数（推荐）**

修改 `detectSensitiveData()` 和 `countSensitiveMatches()` 函数签名：

```typescript
// src/detection/sensitive-detector.ts

/**
 * 检测敏感数据（包含自定义表达式）
 * @param text 待检测文本
 * @param enabledTypes 启用的内置规则类型
 * @param customExpression 自定义逻辑表达式（可选）
 * @returns 敏感类型计数
 */
export function detectSensitiveData(
    text: string, 
    enabledTypes: string[],
    customExpression?: string  // 新增参数
): Record<string, number> {
    // 1. 执行内置规则检测
    const counts = detectBuiltinRules(text, enabledTypes);
    
    // 2. 执行自定义表达式检测（如果提供且非空）
    if (customExpression && customExpression.trim()) {
        try {
            const ast = parseExpression(customExpression);
            const isMatch = evaluate(ast, text);
            if (isMatch) {
                counts['custom_expression'] = 1;
            }
        } catch (error: any) {
            // 表达式解析失败，记录警告但不影响其他检测
            mainLogger.warn(`自定义表达式解析失败: ${error.message}`);
        }
    }
    
    return counts;
}

/**
 * 获取高亮范围（包含自定义表达式匹配）
 */
export function getHighlights(
    text: string, 
    enabledTypes: string[],
    customExpression?: string  // 新增参数
): HighlightRange[] {
    // 1. 获取内置规则的高亮
    const highlights = getBuiltinHighlights(text, enabledTypes);
    
    // 2. 如果自定义表达式匹配，添加特殊高亮（可选）
    if (customExpression && customExpression.trim()) {
        try {
            const ast = parseExpression(customExpression);
            const isMatch = evaluate(ast, text);
            if (isMatch) {
                // 可以选择高亮整个文本，或者不显示高亮仅标记文件
                // 这里选择不添加高亮，仅在文件级别标记
            }
        } catch (error: any) {
            mainLogger.warn(`自定义表达式解析失败: ${error.message}`);
        }
    }
    
    return highlights;
}
```

**方案 B：新增独立函数（备选）**

如果不想修改现有函数签名，可以新增专用函数：

```typescript
/**
 * 评估自定义表达式
 * @param text 待检测文本
 * @param expression 逻辑表达式
 * @returns 是否匹配
 */
export function evaluateCustomExpression(
    text: string, 
    expression: string
): boolean {
    if (!expression || !expression.trim()) {
        return false;
    }
    
    const ast = parseExpression(expression);
    return evaluate(ast, text);
}
```

然后在调用处合并结果：

```typescript

    const isCustomMatch = evaluateCustomExpression(content, customExpression);

    }
}
```

**推荐选择**: **方案 A**，因为：
1. 集中管理所有检测逻辑
2. 避免在多处重复调用
3. 便于后续维护和优化

**结果合并策略**:
- 自定义表达式匹配的文件，在结果中添加 `'custom_expression': 1`
- 前端可以将此类型显示为"自定义规则匹配"
- 可以与内置规则并列显示，或单独分类

### 5. 边界情况处理

#### 5.1 输入验证和错误处理

**前端实时验证**:
- 在工具栏输入框添加 `@input` 事件监听
- 每次输入时调用主进程验证表达式语法
- 使用防抖（debounce）避免频繁 IPC 通信（延迟 300ms）

**前端实现示例**（Vue 组件）:

```vue
<template>
  <div class="custom-expression-input">
    <input 
      v-model="expression" 
      @input="onInput"
      :class="{ 'error': validationError, 'valid': isValid }"
      placeholder="输入敏感词（支持 !、()、&、|）"
    />
    <div v-if="validationError" class="error-message">
      {{ validationError }}
    </div>
    <div v-if="isValid && expression" class="valid-indicator">
      ✓ 语法正确
    </div>
  </div>
</template>

<script setup lang="ts">
import { ref, watch } from 'vue';

const expression = ref('');
const validationError = ref<string>('');
const isValid = ref(false);
let debounceTimer: NodeJS.Timeout | null = null;

// 防抖验证
const onInput = () => {
  if (debounceTimer) clearTimeout(debounceTimer);
  
  debounceTimer = setTimeout(async () => {
    const result = await window.api.validateExpression(expression.value);
    
    if (result.valid) {
      validationError.value = '';
      isValid.value = true;
      
      // 自动保存有效表达式
      await window.api.setCustomExpression(expression.value);
    } else {
      validationError.value = result.error || '语法错误';
      isValid.value = false;
    }
  }, 300);  // 300ms 防抖
};

// 加载已保存的表达式
onMounted(async () => {
  const result = await window.api.getCustomExpression();
  if (result.success) {
    expression.value = result.expression || '';
  }
});
</script>
```

**后端验证逻辑**:
- 在 `parseExpression()` 中捕获语法错误并抛出有意义的错误信息
- 错误信息应包含：
  - 错误类型（括号不匹配、无效运算符、空关键词等）
  - 错误位置（字符索引）
  - 建议修复方式（可选）

**常见错误提示示例**:
```
❌ "括号不匹配：缺少右括号（位置 5）"
❌ "无效的运算符：在位置 8 处发现 '&'，前面缺少关键词"
❌ "空的关键词：在位置 3 处发现连续运算符"
❌ "不支持的字符：在位置 10 处发现 '@'"
✅ "语法正确"
```

**边界情况处理**:
- **空字符串**: 视为禁用自定义表达式，不报错
- **纯空格**: trim 后视为空字符串
- **仅运算符**: 如 `& | !`，报语法错误
- **未闭合括号**: 如 `(密码 & 身份证`，报"括号不匹配"

#### 5.2 性能考虑

- **表达式缓存**：避免重复解析同一表达式
- **短路求值**：`&` 和 `\|` 运算支持短路优化
- **大文件优化**：流式处理时逐行评估，避免加载全部内容

### 9. 注意事项和潜在问题

⚠️ **关键点**：

1. **使用成熟的库**：优先使用 expr-eval 等成熟库
   - 避免重复造轮子
   - 确保解析器的稳定性和正确性
   - 减少维护成本
   - 利用社区的测试和优化成果
   
2. **优先级必须正确**：`( ) > ! > & > |`
   - 如果使用的库支持自定义运算符，需配置正确的优先级
   - expr-eval 默认支持标准运算符优先级
   - 需要在 Parser 中严格实现

3. **错误处理要友好**：清晰的错误提示
   - 前端实时验证，立即反馈
   - 后端捕获异常，返回有意义的错误信息
   - 错误信息包含位置和修复建议

4. **性能要考虑**：避免在大文件上频繁解析
   - **表达式缓存**：同一表达式只解析一次，后续直接使用 AST
   - **短路求值**：`&` 和 `|` 运算支持短路优化
   - **流式处理**：大文件分块处理，避免一次性加载全部内容

5. **兼容性要保证**：不影响现有的敏感词检测功能
   - 自定义表达式是**附加功能**，不替换内置规则
   - 空表达式或解析失败时，降级为仅使用内置规则
   - 保持向后兼容，旧配置文件无此字段时使用默认值

6. **安全性要注意**：防止表达式注入攻击
   - 虽然只是文本匹配，风险较低
   - 但仍需限制表达式长度（如最多 1000 字符）
   - 限制关键词数量（如最多 50 个关键词）
   - 防止正则表达式注入（如果关键词中包含特殊字符）

7. **Worker 线程中的模块导入**：
   - 确保 `expression-parser.ts` 可以在 Worker 环境中运行
   - 避免使用浏览器特有的 API（如 window、document）
   - 测试 Worker 中的 import 是否正常工作

8. **与现有需求的对比**：
   - docs 目录下已有 `需求_增强搜索功能_2026-05-12.md`
   - 需要确认两个需求是否重叠或冲突
   - 如果有重叠，明确优先级和整合方案
   - **当前判断**：增强搜索可能是指文件内容搜索，而本需求是敏感词检测，两者可能互补而非冲突
