**遵循项目规范**:

- ✅ 通过 `registerExtractor` 自动注册

**与 OpenDocument 对比**:
```
相似点:
- 都是 ZIP 压缩包
- 都使用 unzipFile 解压
- 都用正则提取 XML 文本

不同点:
- OFD: Content.xml 在 Pages 目录
- ODF: content.xml 在根目录
- OFD: <TextCode> 标签
- ODF: <text:p> 标签
```

### 2. 文本提取策略

**四层降级机制**:

```
Level 1: <TextCode> 标签
  ↓ (未找到)
Level 2: <CGTransform> 标签
  ↓ (未找到)
Level 3: <Layer> 中文本对象
  ↓ (未找到)
Level 4: 通用文本节点 Fallback
```

**优势**:
- 兼容多种 OFD 变体格式
- 最大化文本提取成功率
- 优雅降级，不会完全失败

### 3. 错误处理

**处理的错误类型**:

| 错误类型 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 文件不存在 | 返回空结果 | "未找到文件" |
| ZIP 解压失败 | 捕获异常 | "文件格式无效" |
| 加密文件 | 检测压缩类型 | "文件已加密" |
| 解析超时 | 30秒自动终止 | "处理超时" |
| 无文本内容 | 返回空字符串 | "未能提取文本" |

## 💡 关键技术决策

### 1. 为什么不使用 jit-viewer？

**原因**:
- ❌ jit-viewer 主要面向预览，不提供文本提取 API
- ❌ 失去对解析过程的控制
- ❌ 可能引入不必要的依赖

**选择自定义解析器**:
- ✅ 完全自主控制
- ✅ 针对扫描场景优化
- ✅ 无额外依赖
- ✅ 易于维护

### 2. 为什么不用 OpenDocument 解析器？

**原因**:
- ❌ OFD 和 ODF 是不同的国家标准
- ❌ XML 结构完全不同
- ❌ 标签名称不同（TextCode vs text:p）

**但可以复用思路**:
- ✅ ZIP 解压逻辑
- ✅ 正则提取方法
- ✅ Extractor 架构

### 3. 为什么选择正则而非 XML 解析器？

**原因**:
- ✅ 性能更好（无需构建 DOM 树）
- ✅ 内存占用更低
- ✅ 容错性更强（ malformed XML 也能处理）
- ✅ 代码更简洁

**权衡**:
- ⚠️ 正则可能不够精确
- ✅ 但对于文本提取场景足够
- ✅ 有四层降级机制保证成功率
