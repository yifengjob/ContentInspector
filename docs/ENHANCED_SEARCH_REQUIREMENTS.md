## 一、背景与目标

### 1.1 现有功能

**已实现的核心功能**:
1. **敏感信息扫描** - 扫描文件内容，检测身份证、手机号、邮箱等敏感数据
2. **结果列表简单搜索** - 在 ResultsTable 中通过文件路径字符串匹配过滤结果

**当前局限性**:
- ❌ 结果列表搜索仅支持简单字符串匹配
- ❌ 不支持布尔逻辑运算符（AND、OR、NOT）
- ❌ 仅在文件名/路径中搜索，无法搜索文件内容
- ❌ 需要先完成扫描才能搜索，无法在扫描过程中实时过滤

### 1.2 增强目标

需要在现有基础上增加强大的搜索能力：

1. ✅ **保留现有功能** - 不改变敏感信息扫描和结果列表搜索的核心逻辑
2. ✅ **增强结果列表搜索** - 支持布尔逻辑表达式（AND、OR、NOT、分组）
3. ✅ **新增文件内容搜索** - 在工具栏添加强大的表达式搜索功能，搜索文件内容
4. ✅ **单次遍历优化** - 敏感检测和内容搜索在一次文件遍历中完成
5. ✅ **统一语法** - 两种搜索使用相同的布尔表达式语法

## 二、功能规格

### 2.1 两种搜索功能对比

| 特性 | 结果列表搜索（增强） | 文件内容搜索（新增） |
|------|-------------------|-------------------|
| **位置** | ResultsTable 表头 | 主工具栏右侧 |
| **搜索范围** | 文件名/路径 | 文件内容 |
| **搜索时机** | 扫描完成后过滤 | 扫描过程中实时匹配 |
| **当前能力** | 简单字符串匹配 | - |
| **增强后能力** | 布尔逻辑表达式 | 布尔逻辑表达式 |
| **性能影响** | 无（内存过滤） | 小（单次遍历中完成） |
| **适用场景** | 快速定位已知文件 | 查找包含特定内容的文件 |

**重要原则**: 两种搜索功能并存，使用相同的布尔表达式语法，保持一致的用户体验。

### 2.2 布尔逻辑运算符（通用）

两种搜索都支持以下运算符：

| 运算符 | 含义 | 优先级 | 示例 | 说明 |
|--------|------|--------|------|------|
| `()` | 分组 | 最高 | `(A \| B) & C` | 改变运算顺序 |
| `!` | 非(NOT) | 高 | `!测试` | 排除匹配项 |
| `&` | 与(AND) | 中 | `密码 & 银行卡` | 同时满足 |
| `\|` | 或(OR) | 低 | `身份证 \| 手机号` | 任一满足 |

**运算符优先级**: `()` > `!` > `&` > `|`

### 2.3 文件内容搜索开关逻辑

#### 模式 A：开关打开（默认）

**行为**: 单次文件遍历中同时执行两项任务
- 敏感信息检测（身份证、手机号等）
- 文件内容关键词搜索（布尔表达式匹配）

**结果**: 两个结果集取并集（OR关系），去重后显示

**优势**: 功能完整，一次扫描获得全部信息

#### 模式 B：开关关闭

**行为**: 仅执行文件内容关键词搜索，跳过敏感检测
- 不执行敏感信息扫描
- 仅根据表达式在文件内容中搜索关键词

**优势**: 速度更快，适合只需要查找特定内容的场景

### 2.4 搜索示例

#### 结果列表搜索（路径匹配）

```bash
# 基础搜索
合同                    # 文件名包含"合同"
财务 报告               # 文件名同时包含"财务"和"报告"

# 逻辑运算符
合同 | 协议             # 文件名包含"合同"或"协议"
2024 & 总结             # 文件名同时包含"2024"和"总结"
!草稿                   # 文件名不包含"草稿"

# 分组
(财务 | 预算) & 2024    # 文件名包含("财务"或"预算")且包含"2024"

# 实际场景
合同 | 协议 | 合约      # 查找所有合同相关文件
2024 & 财务 & 报告      # 查找2024年的财务报告
报告 & !临时 & !草稿    # 排除临时文件
```

#### 文件内容搜索（内容匹配）

```bash
# 基础搜索
机密                    # 文件内容包含"机密"
密码 银行卡             # 文件内容同时包含"密码"和"银行卡"

# 逻辑运算符
身份证 | 手机号         # 文件内容包含"身份证"或"手机号"
密码 & 银行卡           # 文件内容同时包含"密码"和"银行卡"
!测试                   # 文件内容不包含"测试"

# 分组
(财务 | 预算) & 2024    # 文件内容包含("财务"或"预算")且包含"2024"

# 复杂组合
(密码 | 银行卡) & !测试 # 包含("密码"或"银行卡")且不包含"测试"

# 实际场景
机密                    # 查找机密文档（开关打开时同时检测敏感信息）
财务 & 报告 & 2024      # 查找财务报告（开关关闭时快速搜索）
项目计划 & !草稿        # 排除草稿文件
(身份证 | 手机号) & 合同 # 查找敏感信息相关的合同文件（开关打开）
```

## 四、技术架构

### 4.1 核心设计原则

1. **单次遍历** - 开关打开时，在同一次文件遍历中同时完成敏感检测和关键词匹配
2. **复用解析器** - 两种搜索使用相同的表达式解析器，保证语法一致
3. **向后兼容** - 不改变现有敏感信息扫描的核心逻辑
4. **优雅降级** - 表达式解析失败时退回到简单字符串匹配

### 4.2 系统架构

```
┌─────────────────────────────────────────────┐
│                  前端 (Vue)                   │
│                                              │
│  ┌────────────────┐    ┌─────────────────┐  │
│  │ 结果列表搜索    │    │ 文件内容搜索     │  │
│  │ (ResultsTable) │    │ (工具栏)         │  │
│  │                │    │                  │  │
│  │ 布尔表达式过滤  │    │ 开关 + 表达式    │  │
│  └────────────────┘    └────────┬────────┘  │
│                                 │            │
└─────────────────────────────────┼────────────┘
                                  │ IPC
┌─────────────────────────────────┼────────────┐
│              后端 (Electron)     │            │
│                                 │            │
│  ┌──────────────────────────────▼────────┐  │
│  │   文件遍历引擎 (scanner.ts)           │  │
│  │   【原有逻辑，保持不变】               │  │
│  └──────────────┬───────────────────────┘  │
│                 │                           │
│  ┌──────────────▼───────────────────────┐  │
│  │   Worker Pool (file-worker.ts)       │  │
│  │                                      │  │
│  │  1. 提取文件文本内容                  │  │
│  │  2. 敏感信息检测（条件执行）          │  │
│  │  3. 关键词表达式匹配（始终执行）      │  │
│  │  4. 合并结果                         │  │
│  └──────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

### 4.3 关键技术点

#### 表达式解析器
- 需要实现一个通用的布尔表达式解析器
- 支持词法分析、语法分析、AST构建
- 前后端共用，保证语法一致性

#### Worker 线程处理
- 在现有 file-worker.ts 中嵌入关键词匹配逻辑
- 根据开关决定是否执行敏感检测
- 始终执行关键词匹配（只要有表达式）
- 合并两种结果并返回

#### 结果列表过滤
- 在前端 ResultsTable 组件中使用相同的解析器
- 对扫描结果进行布尔表达式过滤
- 同时匹配文件名和完整路径
- 解析失败时退回到简单字符串匹配

## 五、性能要求

### 5.1 性能指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 小文件 (<5MB) 处理时间增加 | < 20% | 相比原敏感扫描 |
| 内存使用峰值增加 | < 50MB | 关键词匹配的额外开销 |
| 表达式解析时间 | < 10ms | 前端实时解析 |
| 结果列表过滤响应 | < 100ms | 1000条结果的过滤时间 |

### 5.2 优化策略

1. **单次遍历** - 避免重复读取和解析文件
2. **内存匹配** - 关键词匹配在内存中进行，开销小
3. **并发控制** - 复用现有的并发机制
4. **缓存机制** - 可考虑缓存解析后的AST
