## 一、问题分析

### 1.1 当前状况

✅ **jit-viewer 支持 OFD 预览**
- jit-viewer v1.1+ 已支持 OFD 格式预览（国产办公文档格式）
- 支持在浏览器中渲染 OFD 文档
- 纯前端实现，无需后端服务

❌ **项目缺少 OFD 文本提取能力**
- 当前扫描功能不支持 `.ofd` 文件
- 无法从 OFD 文件中提取文本进行敏感词检测
- 仅能预览，不能扫描内容

### 1.2 核心问题

> **用户问题**: "如果 jit-viewer 支持解析 OFD 提取 OFD 中的文本，就用 jit-viewer 解析，不支持是否可以用 opendocument 或 xml 解析器进行解析？"

**关键区分**：
- **预览（Preview）** ≠ **文本提取（Text Extraction）**
- jit-viewer 主要用于**视觉渲染**，不一定提供**文本提取 API**
- 扫描功能需要的是**纯文本内容**，用于敏感词匹配

## 二、OFD 格式技术分析

### 2.1 OFD 格式特点

**OFD (Open Fixed-layout Document)** 是中国国家标准的版式文档格式：

| 特性 | 说明 |
|------|------|
| **本质** | 基于 XML 的 ZIP 压缩包（类似 ODF/DOCX） |
| **结构** | 包含 `OFD.xml`、`Pages/`、`Content/` 等目录 |
| **文本存储** | 文本以 XML 标签形式存储在页面描述文件中 |
| **优势** | 结构化程度高，便于机器解析和语义提取 |
| **对比 PDF** | 比 PDF 更容易解析，所有元素都是明文 XML |

### 2.2 OFD 文件结构示例

```
example.ofd (ZIP 压缩包)
├── OFD.xml                    # 文档元数据
├── Doc_0/
│   ├── Document.xml           # 文档结构定义
│   ├── Pages/
│   │   ├── Page_0/
│   │   │   ├── Content.xml    # 页面内容（文本在此）
│   │   │   └── Res.xml        # 资源引用
│   │   └── Page_1/
│   │       └── ...
│   └── PublicRes.xml          # 公共资源
└── Annotations/               # 批注（可选）
```

**文本位置**: `Doc_0/Pages/Page_*/Content.xml`

**XML 示例**:
```xml
<Page xmlns="http://www.ofdspec.org/2016">
  <Layer Type="Foreground">
    <TextObject>
      <FillColor Value="0,0,0"/>
      <TextCode>这是 OFD 中的文本内容</TextCode>
    </TextObject>
  </Layer>
</Page>
```

#### 可行性分析

**jit-viewer 的能力**：
- ✅ 支持 OFD **预览渲染**
- ❓ 是否提供**文本提取 API** - **需要验证**

**调研结果**：
根据公开文档，jit-viewer 主要定位为**预览引擎**，未明确提供文本提取接口。即使内部有解析能力，也未对外暴露 API。

#### 优缺点分析

| 优点 | 缺点 |
|------|------|
| ✅ 统一技术栈（与预览共用） | ❌ jit-viewer 可能不提供文本提取 API |

**不推荐作为首选方案**，原因：
1. jit-viewer 主要面向预览，文本提取不是其核心功能
2. 即使支持，API 可能不稳定或不完善
3. 失去对解析过程的控制权

### 3.2 方案 B：自定义 OFD 解析器（推荐度：⭐⭐⭐⭐⭐）

#### 技术原理

OFD 本质是 ZIP + XML，可以复用现有的 OpenDocument 解析架构：

```
OFD 文件 → unzip 解压 → 遍历 XML 文件 → 正则提取文本 → 合并返回
```

#### 技术实现

**Step 1: 创建 OFD 提取器**

```typescript
/**
 * OFD 文件提取器 - 使用 fflate 解压 + XML 解析
 * 支持: ofd
 */

import { unzipFile, extractEntriesText } from '../../utils/zip-utils';
import type { ExtractorResult } from '../types';
import { BaseExtractor } from '../base-extractor';
import { withTimeout, withLogging, composeDecorators } from '../extractor-decorators';

class OfdExtractor extends BaseExtractor {
  constructor() {
    super({
      name: 'OfdExtractor',
      verboseLogging: false,
    });
  }

  protected async doExtract(filePath: string): Promise<ExtractorResult> {
    try {
      // 1. 解压 OFD 文件（ZIP 格式）
      const entries = await unzipFile(filePath);

      // 2. 查找所有 Content.xml 文件（页面内容）
      const contentFiles = entries.filter((e) => 
        e.name.includes('/Content.xml') || e.name.endsWith('Content.xml')
      );

      if (contentFiles.length === 0) {
        this.logger.warn('[OfdExtractor] 未找到 Content.xml 文件');
        return this.buildResult('', this.config.name);
      }

      // 3. 提取所有页面的文本
      const textChunks: string[] = [];

      for (const contentEntry of contentFiles) {
        const xmlContent = extractEntriesText([contentEntry])[0];
        if (!xmlContent) continue;

        // 4. 使用正则表达式提取文本
        const pageTexts = this.extractTextFromXml(xmlContent);
        textChunks.push(...pageTexts);
      }

      // 5. 合并所有文本
      const allText = textChunks.join('\n');
      return this.buildResult(allText, this.config.name);
    } catch (error: any) {
      if (error.message.includes('unknown compression type')) {
        this.logger.warn('[OfdExtractor] OFD 文件可能已加密或损坏');
      } else {
        this.logger.error(`[OfdExtractor] 解析失败: ${error.message}`);
      }

      return this.handleError(error, filePath);
    }
  }

  /**
   * 从 OFD XML 中提取文本
   * 
   * OFD 文本通常存储在 <TextCode> 或 <CTM> 标签中
   */
  private extractTextFromXml(xmlContent: string): string[] {
    const textChunks: string[] = [];

    // 方法 1: 提取 <TextCode> 标签中的文本（最常见）
    const textCodeMatches = xmlContent.match(/<TextCode[^>]*>(.*?)<\/TextCode>/gs);
    if (textCodeMatches) {
      for (const match of textCodeMatches) {
        const text = match.replace(/<[^>]+>/g, '').trim();
        if (text) {
          textChunks.push(text);
        }
      }
    }

    // 方法 2: 提取 <CGTransform> 中的文本（某些 OFD 变体）
    if (textChunks.length === 0) {
      const cgMatches = xmlContent.match(/<CGTransform[^>]*>(.*?)<\/CGTransform>/gs);
      if (cgMatches) {
        for (const match of cgMatches) {
          const text = match.replace(/<[^>]+>/g, '').trim();
          if (text) {
            textChunks.push(text);
          }
        }
      }
    }

    // 方法 3: 通用 fallback - 提取所有非空文本节点
    if (textChunks.length === 0) {
      const allTextNodes = xmlContent.match(/>([^<>]+)</g);
      if (allTextNodes) {
        for (const node of allTextNodes) {
          const text = node.slice(1, -1).trim();
          if (text && text.length > 1) {  // 过滤单个字符（可能是符号）
            textChunks.push(text);
          }
        }
      }
    }

    return textChunks;
  }
}

// 创建实例并应用装饰器
const ofdExtractor = new OfdExtractor();

const enhancedOfdExtract = composeDecorators(ofdExtractor.extract.bind(ofdExtractor), [
  (fn) => withTimeout(fn, { timeoutMs: 30000 }),  // 30秒超时
  (fn) => withLogging(fn, { logError: true, prefix: 'OfdExtractor' }),
]);

/**
 * 提取 OFD 文件内容
 * @param filePath 文件路径
 * @returns 提取结果
 */
export async function extractOfd(filePath: string): Promise<ExtractorResult> {
  return await enhancedOfdExtract(filePath);
}
```

### 3.3 方案 C：使用第三方 OFD 解析库（推荐度：⭐⭐⭐）

#### 可用库调研

| 库名 | 语言 | 成熟度 | 备注 |
|------|------|--------|------|
| **ofdrw** | Java | ⭐⭐⭐⭐⭐ | 最成熟的 OFD 库，但需要 JVM |
| **libofd** | C++ | ⭐⭐⭐⭐ | 底层库，需要编译 |
| **pyofd** | Python | ⭐⭐⭐ | Python 生态，需桥接 |
| **ofd-parser-js** | JavaScript | ⭐⭐ | NPM 上有几个小库，但不成熟 |

#### 技术实现（以 ofd-parser-js 为例）

```typescript
import { parseOfd } from 'ofd-parser-js';

async function extractOfdText(filePath: string): Promise<string> {
  try {
    const document = await parseOfd(filePath);
    
    // 提取所有页面的文本
    const texts = document.pages.map(page => 
      page.textObjects.map(obj => obj.content).join(' ')
    );
    
    return texts.join('\n');
  } catch (error) {
    throw new Error(`OFD 解析失败: ${error.message}`);
  }
}
```

#### 优缺点分析

| 优点 | 缺点 |
|------|------|
| ✅ 可能有更完善的解析逻辑 | ❌ JavaScript 生态中 OFD 库不成熟 |
| ✅ 减少自己维护的成本 | ❌ 增加依赖，可能引入安全问题 |
| | ❌ 库可能停止维护 |
| | ❌ 不一定适合扫描场景（可能过度解析） |

## 五、技术细节

### 5.1 OFD XML 文本提取策略

**优先级顺序**：

1. **精确匹配** `<TextCode>` 标签
   ```regex
   /<TextCode[^>]*>(.*?)<\/TextCode>/gs
   ```

2. **备选匹配** `<CGTransform>` 标签
   ```regex
   /<CGTransform[^>]*>(.*?)<\/CGTransform>/gs
   ```

3. **通用 Fallback** 提取所有文本节点
   ```regex
   />([^<>]+)</g
   ```

### 5.2 文件大小限制

参考 OpenDocument 配置：

```typescript
maxSizeMB: 50  // OFD 文件通常比 PDF 小，50MB 足够
```

### 5.3 超时设置

```typescript
withTimeout(fn, { timeoutMs: 30000 })  // 30秒超时
```

### 5.4 错误处理

| 错误类型 | 处理方式 |
|---------|---------|
| 文件损坏 | 返回空文本，记录警告日志 |
| 加密文件 | 返回空文本，提示"文件已加密" |
| 解析超时 | 抛出超时异常，由调度器处理 |
| 未知格式 | 尝试 Fallback 方法，仍失败则返回空 |

## 七、为什么不使用 OpenDocument 解析器？

### 7.1 格式差异

虽然 OFD 和 ODF 都是 ZIP + XML，但**内部结构完全不同**：

| 特性 | OFD | ODF (ODT/ODS/ODP) |
|------|-----|-------------------|
| **标准** | 中国国家标准 (GB/T 33190-2016) | 国际标准 (ISO/IEC 26300) |
| **根文件** | `OFD.xml` | `META-INF/manifest.xml` |
| **内容文件** | `Doc_0/Pages/*/Content.xml` | `content.xml` |
| **文本标签** | `<TextCode>` | `<text:p>`, `<text:h>` |
| **命名空间** | `http://www.ofdspec.org/2016` | `urn:oasis:names:tc:opendocument:xmlns:*` |

### 7.2 结论

**不能直接复用 OpenDocument 解析器**，原因：
1. XML 结构和标签完全不同
2. 命名空间不同
3. 文本存储方式不同

**但可以复用解析思路**：
- ZIP 解压逻辑（`zip-utils`）
- XML 文本提取方法（正则表达式）
- 提取器架构（BaseExtractor + 装饰器）
